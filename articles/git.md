# 版本管理

## 提交前缀

| 类型 | 前缀 | 图标 | 说明 |
|------|------|------|------|
| 初始化 | init | 🎉 | 项目或模块的初次提交。 |
| 新增 | feat | ✨ | 新增功能。发布新版本时，次版本号递增。 |
| 移除 | remove/del | ➖ | 删除文件、功能或代码。 |
| 修复 | fix | 🐛 | 修复Bug。发布新版本时，修订号递增。 |
| 文档 | docs | 📚 | 仅文档更新（如README、注释、帮助文档）。 |
| 格式 | style | 💄 | 代码格式调整（不影响代码逻辑，如空格、缩进、分号）。 |
| 重构 | refactor | ♻️ | 代码重构（非新增功能，也非修复Bug的结构性优化）。 |
| 优化 | optimize | ⚙️ | 一般性代码优化（比"重构"更小的改动）。 |
| 性能 | perf | ⚡ | 提升性能的代码更改。 |
| 测试 | test | ✅ | 增加或修改测试用例（不涉及生产代码）。 |
| 配置 | config | ⚙️ | 配置文件更改。 |
| 合并 | merge | 🔀 | 合并分支或代码。 |
| 更新 | update | ⬆️ | 普通更新（如更新文档内容、更新依赖版本）。 |
| 杂项 | chore | 🔧 | 其他杂项更新（如构建流程、工具链、依赖管理）。 |
| 构建 | build | 📦 | 构建系统或项目依赖的更改（Webpack、npm脚本、外部包）。 |
| 回退 | revert | ⏪️ | 撤销之前的某次提交。 |

## 想做更新日志？

### Github原生发布页面Release

1. 进入 GitHub 仓库页面
2. 点击右侧 "Releases"
3. 点击 "Draft a new release"
4. 填写：
   - Tag version: v1.0.0
   - Release title: 版本名称
   - 详细描述（支持 Markdown）

# Markdown基本格式

Markdown 通过简单的符号组合，就能实现丰富的文字样式和文档结构。其核心格式可分为以下几类：

## 文字样式与强调

- **加粗**：用两个星号或下划线包裹文字，如 `**文本**` 或 `__文本__`，效果：**文本**。
- *斜体*：用一个星号或下划线包裹文字，如 `*文本*` 或 `_文本_`，效果：*文本*。
- ~~删除线~~：用两个波浪号包裹文字，如 `~~文本~~`，效果：~~文本~~。
- <sup>上</sup><sub>下</sub>标：用 `<sup>` 和 `<sub>` HTML标签实现，如 X<sup>2</sup> 和 H<sub>2</sub>O。

## 结构组织

- **标题**：使用 1 到 6 个 `#` 开头，对应一级到六级标题，如 `# 一级标题`。
- **列表**：
  - 无序列表：用 `-`、`+` 或 `*` 开头。
  - 有序列表：用数字加 `.` 开头，如 `1. 第一项`。
- **引用块**：用 `>` 开头，用于引述内容。
- **分割线**：单独一行使用三个或以上的 `---`、`***` 或 `___`。

## 链接与媒体

- **链接**：直接输入网址，如 https://bilibili.com。或通过超链接的方式附加，如[bilibili](https://bilibili.com)（`[bilibili](https://bilibili.com)`）。
- **图片**：在上述超链接格式的最前方添加`!`，并在url中填入图片文件的相对地址，如![texture](texture.jpg)（`![texture](texture.jpg)`）。
- **脚注 / 代码**：用 `` ` ` `` 在文中标记，在文档底部用 `: 注释内容` 说明。

## 代码与高级排版

- **行内代码**：用一个反引号包裹，如 `` `代码` ``。
- **代码块**：用三个反引号包裹，并可选指定语言以实现语法高亮。
- **表格**：使用竖线 `|` 和连字符 `-` 创建。

| 表头1 | 表头2 |
|-------|-------|
| 内容1 | 内容2 |

- **任务列表**：用 `- [ ]` 表示未完成，`- [x]` 表示已完成。
